# Сниффинг мобильного API Zara с iPhone (mitmproxy)

Цель: поймать реальные запросы приложения Zara к бэкенду наличия товара, чтобы
использовать этот эндпоинт в `monitor.py` вместо веб-эндпоинта, который блокируется
антибот-защитой.

mitmproxy — это разовый инструмент для исследования, он не входит в зависимости
проекта и не нужен в рантайме.

## 0. Требования

- Mac и iPhone в одной Wi-Fi сети.
- Установленное приложение Zara на iPhone, залогиниться необязательно.

## 1. Установить mitmproxy на Mac

```bash
brew install mitmproxy
```

Запустить веб-интерфейс (удобнее читать трафик, чем в консольном UI):

```bash
mitmweb
```

По умолчанию поднимется прокси на порту `8080` и откроется браузер с UI на
`http://127.0.0.1:8081`.

Если хочешь консольный вариант вместо веб — используй `mitmproxy` вместо `mitmweb`,
дальнейшие шаги те же.

## 2. Узнать IP Mac в локальной сети

```bash
ipconfig getifaddr en0
```

Если Mac подключён по Wi-Fi через другой интерфейс — попробуй `en1` и т.д., либо
посмотри в Системных настройках → Wi-Fi → Details → IP-адрес.

## 3. Настроить прокси на iPhone

На iPhone:

1. Настройки → Wi-Fi → нажать (i) у активной сети.
2. Прокрутить вниз до "Настроить прокси" → выбрать "Вручную".
3. Сервер: IP Mac из шага 2. Порт: `8080`.
4. Сохранить.

С этого момента весь HTTP(S)-трафик с телефона идёт через mitmproxy на Mac —
если ничего не сниффишь, лучше выключить прокси обратно (иначе интернет на
телефоне может пропасть при выключенном mitmproxy на Mac).

## 4. Установить и довериться сертификату mitmproxy на iPhone

mitmproxy — это MITM-прокси для HTTPS, поэтому телефон должен доверять его
корневому сертификату, иначе приложения с pinning или системный Safari будут
рвать соединение.

1. На iPhone открыть Safari и зайти на `http://mitm.it` (именно http, не https).
2. Выбрать "Apple" внизу страницы, скачать профиль сертификата.
3. Настройки → профиль появится в самом верху (под именем аккаунта) →
   "Установить" → ввести код-пароль → "Установить" ещё раз.
4. **Важно, отдельный шаг**: Настройки → Основные → Об этом устройстве →
   Доверенные сертификаты (Certificate Trust Settings) → включить полное доверие
   для "mitmproxy" сертификата.

Без шага 4 сертификат установлен, но не является доверенным для TLS — HTTPS
трафик не расшифруется.

## 5. Проверить, что трафик идёт через mitmproxy

Открыть в Safari на iPhone любой сайт (например `https://example.com`) и
посмотреть, что запрос появился в mitmweb UI на Mac. Если появился и открылся
без ошибок — всё настроено верно.

## 6. Поймать трафик приложения Zara

1. В mitmweb можно включить фильтр по домену, чтобы не тонуть в остальном
   трафике телефона, например: `~d zara`.
2. Открыть приложение Zara на iPhone.
3. Зайти в раздел с товарами, открыть конкретный товар (лучше тот же артикул,
   что уже используется в `.env` → `ZARA_PRODUCT_ID`).
4. На странице товара переключить/открыть выбор размера — это должно
   спровоцировать запрос за детальной информацией о наличии.
5. Полистать между несколькими размерами/цветами, чтобы поймать несколько
   похожих запросов и увидеть, какие параметры в URL/теле меняются, а какие
   статичны.

## 7. Что искать в пойманных запросах

Ищи запрос(ы), в ответе которых есть данные о размерах и наличии
(`availability`, `stock`, `sizes` и т.п. — по аналогии со структурой, которую
уже разбирает `parse_sizes()` в `monitor.py`).

Обычно у мобильных бэкенд-доменов Zara/Inditex другой хост, чем у сайта
(`www.zara.com`) — например поддомены вида `api.zara.net`,
`www.zara.net`, `itxrest.zara.net` (конкретный домен нужно смотреть в
реальном трафике, он может отличаться по региону/версии приложения).

Для найденного запроса зафиксируй и сохрани:

1. **Полный URL**, включая путь — отметь, какие сегменты являются
   `product_id`/`store_id`/`country`, а какие статичны.
2. **HTTP-метод** (GET/POST) и **тело запроса**, если это POST.
3. **Все заголовки запроса**, особенно нестандартные: значения вида
   `Ocp-Apim-Subscription-Key`, `x-api-key`, `Authorization`,
   `User-Agent` мобильного приложения, `x-device-id`, `x-version` и подобные —
   именно их обычно требует бэкенд и именно они блокируют запрос при
   отсутствии.
4. **Тело/структуру ответа** — JSON с наличием размеров, чтобы понять, как
   переписать `parse_sizes()` под новую структуру.
5. Обрати внимание, требует ли эндпоинт **авторизации/сессии** (токен,
   привязанный к пользователю/устройству) — если да, такой эндпоинт не подойдёт
   для анонимного мониторинга без логина, нужно искать публичный вариант
   (обычно есть отдельный "guest"/анонимный эндпоинт для отображения каталога).

Экспортировать найденный запрос из mitmweb можно через "Export" → "curl command"
(значок в интерфейсе рядом с запросом) — это удобно принести обратно в чат для
переноса в `monitor.py`.

## 8. Обязательно выключить прокси после сниффинга

На iPhone: Настройки → Wi-Fi → (i) → Настроить прокси → "Выключено".
Иначе телефон продолжит пытаться слать трафик через Mac, даже когда mitmproxy
не запущен, и интернет на телефоне перестанет работать.

Опционально: если сертификат mitmproxy больше не нужен, можно удалить профиль
в Настройки → Основные → VPN и управление устройством → mitmproxy → Удалить
профиль.

## 9. Что делать с результатом

Принеси в проект (или прямо в диалог с Claude):

- URL эндпоинта и метод
- Заголовки запроса (значения секретных ключей можно частично замаскировать,
  но структуру и названия заголовков нужно сохранить полностью)
- Пример JSON-ответа

Дальше это используется для замены `PRODUCT_API_URL` и `HEADERS` в
`monitor.py:25-38` и, при необходимости, переписывания `parse_sizes()`
(`monitor.py:44-65`) под новую структуру ответа.
